# Web 自动化脚本生成 Agent：交接完成后的五阶段计划书

更新时间：2026-06-22

## 0. 当前总判断

项目目标不是只让某一个网站跑通，而是实现一个可以根据自然语言任务生成、执行、验证、调试并修复 Web 自动化脚本的 Agent。核心路线仍按五阶段推进：

1. 视觉 fallback 后的稳定 selector 回写。
2. Capture/Replay 行为一致性与性能快路径。
3. 统一 Failure Context 与规则优先 FailureClassifier。
4. 规则优先 ScriptFixer。
5. 带预算、停止条件与升级策略的 DebugLoop。

当前进度：

- 第一阶段：已完成。
- 第二阶段：已完成。
- 第三阶段：已完成。
- 第四阶段：未完成，下一步开始。
- 第五阶段：未完成，依赖第四阶段。

## 1. 新增硬约束

### 1.1 规则必须具有普适性

所有规则、分类、修复策略都必须优先抽象成跨网站通用能力，不能写成只对单一网站、单一按钮、单一 DOM 结构有效的特例。

规则至少要覆盖两类场景：

1. 大部分视频制作/剪辑类网站：
   - 创建项目。
   - 上传视频、图片、音频。
   - 确认素材进入编辑器或时间线。
   - 添加文字、修改文字样式、确认画布可读。
   - 导出低清晰度视频。
   - 删除导出产物或清理项目。

2. 其他普通网站：
   - 表单填写。
   - 文件上传。
   - 搜索/筛选。
   - 登录/认证等待。
   - 动态加载页面。
   - tab、modal、dropdown、toast、列表、分页等常见 UI。

允许有站点适配提示，但必须满足：

- 站点规则只能作为弱 hint，不能覆盖通用判断。
- 不能因为某个站点曾经成功，就把具体 selector、具体文案、具体坐标写死成核心规则。
- 修复策略必须以失败证据、状态变化、postcondition、候选对比为依据。
- 如果规则只适用于一个网站，必须标注为 site hint，并且不能作为主路径验收标准。

## 3. 未完成阶段计划

### 第四阶段：规则优先 ScriptFixer

目标：

对高置信机械失败生成最小 patch。规则能修的先规则修，规则不能修或低置信才允许 LLM 局部 patch。

核心原则：

- patch 必须最小。
- patch 必须保留原 postcondition。
- patch 不能删除断言。
- patch 不能把失败任务改成假成功。
- patch 必须写明：
  - 原失败证据。
  - 分类结果。
  - 修改范围。
  - patch 理由。
  - 预期 postcondition。
  - 回滚信息。

第一批规则修复范围：

1. `selector_miss`
   - 从 captured action 的 fallback selectors、候选快照、失败现场重新选择 selector。
   - 可以增加 selector wait。
   - 只能修改失败 action 的 selector 或等待逻辑。

2. `selector_not_unique`
   - 收窄 selector。
   - 加 role/text/aria/index 约束。
   - 禁止随便点击第一个，必须有稳定选择理由。

3. `postcondition_failed`
   - 补 targeted state wait。
   - 修正动作顺序。
   - 保留原 postcondition。

4. `network_or_load_delay`
   - 用状态驱动 wait 替代固定 sleep。
   - 针对 URL、timeline、素材、导出完成、toast 消失等状态等待。

5. `visual_mapping_untrusted`
   - 重新请求视觉 grounding。
   - 要求 visible label、坐标范围、候选对齐证据。
   - 证据不足时不得回写 selector。

6. `wrong_target`
   - 根据 before/after diff 和 postcondition 判断是否点错目标。
   - 优先重新 ground 目标，而不是重复点击同一 selector。

7. `auth_required` / `security_challenge`
   - 不 patch 脚本业务逻辑。
   - 进入人工登录/验证等待。
   - 验证清除后从失败点继续。

8. `task_unsatisfiable`
   - 不生成成功 patch。
   - 输出不可满足报告。
   - 附证据和已尝试能力。

第四阶段最小验收顺序：

1. 简单表单 selector miss：
   - 注入坏 selector。
   - 分类为 `selector_miss`。
   - 生成 `patched_script_01.py` 和 `patch_01.diff`。
   - 重跑通过原 success assertion。
   - 不调 LLM。

2. 123Apps timeline wait/postcondition：
   - 注入 timeline postcondition 失败或等待不足。
   - 分类为 `postcondition_failed` 或 `network_or_load_delay`。
   - 生成 targeted wait patch。
   - 重跑后 timeline assertion 通过。

3. 视觉 fallback mapping untrusted：
   - 注入视觉坐标越界或 label 不可信。
   - 分类为 `visual_mapping_untrusted`。
   - 生成重新 grounding 或 selector 证据补强策略。
   - 证据不足时保持失败，不假成功。

4. 123Apps 多媒体任务局部能力：
   - 视频、图片、音频三类上传分别能被识别。
   - 至少能对三类素材存在性建立 postcondition。
   - 不要求一开始就完整导出删除，但必须为完整任务铺好状态断言。

### 第五阶段：带预算、停止条件与升级策略的 DebugLoop

目标：

形成完整自动修复闭环：

```text
run generated script
collect failure context
classify
build fix request
apply minimal patch
rerun
compare original postconditions
success / next attempt / LLM escalation / stop
```

每轮必须留下：

- `debug_attempt_01.json`
- `failure_context_01.json`
- `classification_01.json`
- `fix_request_01.json`
- `patch_01.diff`
- `patched_script_01.py`
- `script_run_01.json`
- `postcondition_results_01.json`
- `timings_01.json`
- `llm_budget_01.json`

停止规则：

1. 同一 category + 同一证据指纹重复出现时停止。
2. 同一 patch 失败后不能无限重复应用。
3. postcondition 失败时，即使脚本退出码为 0，也不能算成功。
4. LLM patch 次数用尽必须停止并解释。
5. 认证/安全挑战进入人工等待，不消耗普通修复预算。
6. 不可满足任务禁止被修成假成功。

预算：

- 最大 debug attempts。
- 最大 selector retry。
- 最大 visual grounding。
- 最大 visual verification。
- 最大 LLM patch。
- 最大总耗时。
- 最大单步等待。

第五阶段真实验收：

1. 简单表单：
   - 人工破坏 selector。
   - DebugLoop 自动局部修复。
   - 原断言通过。

2. 123Apps 上传时间线：
   - 注入等待不足或旧 selector。
   - 修复后 timeline assertion 仍通过。

3. 123Apps 多媒体时间线：
   - 上传视频、图片、音频。
   - 三类素材同时存在。
   - 导出最低清晰度。
   - 删除导出视频。
   - 修复过程中不得删除三类素材 postcondition。

4. 视觉 fallback：
   - selector 回写失效时先尝试 fallback selectors。
   - 证据不足才升级视觉。
   - 修复后仍做状态变化和最终视觉验收。

5. Viggle：
   - 登录/OAuth 需要人工时等待用户。
   - 上传视频和图片。
   - 页面接受素材后继续。
   - 不因登录页面误判成 selector miss。

6. 不可满足场景：
   - 输出结构化失败。
   - 不生成伪成功脚本。

## 4. 跨站点通用规则设计

### 4.1 上传规则

上传不能只针对某个站点的按钮文案。通用识别应结合：

- `input[type=file]`
- button/link 文案：
  - Upload
  - Add media
  - Add file
  - Import
  - Choose file
  - Select file
  - 上传
  - 添加媒体
- aria label。
- dropzone。
- accept 属性：
  - `video/*`
  - `image/*`
  - `audio/*`
- 页面状态变化：
  - 文件名出现。
  - 素材卡出现。
  - timeline clip 出现。
  - track/layer 出现。
  - upload progress 结束。

### 4.2 视频制作类网站素材 postcondition

不要只认某个 DOM 类名。应建立多证据判断：

视频素材：

- 文件扩展名：`.mp4`, `.mov`, `.webm`, `.avi`。
- 页面文本出现文件名或截断文件名。
- timeline 中出现 clip/card/canvas/track item。
- 视频相关控件出现：
  - trim
  - crop
  - speed
  - export
  - timeline

图片素材：

- 文件扩展名：`.jpg`, `.jpeg`, `.png`, `.webp`。
- image layer、image card、素材库卡片、画布图层变化。
- 页面中出现 image/photo/picture/layer 等语义。

音频素材：

- 文件扩展名：`.mp3`, `.m4a`, `.wav`, `.aac`, `.ogg`。
- audio track、waveform、music/sound 轨道。
- timeline 中独立音频轨或音频素材卡。

三类同时存在：

- 不要求 DOM 同时暴露三个文件名。
- 可以用类型 postcondition 汇总：
  - `media_type_present: video`
  - `media_type_present: image`
  - `media_type_present: audio`
  - `all_media_types_present: ["video", "image", "audio"]`

### 4.3 导出规则

低清晰度导出不能只按 123Apps 文案写死。通用识别：

- Export / Download / Render / Save / 导出 / 下载。
- 清晰度选项：
  - lowest
  - low
  - 360p
  - 480p
  - SD
  - 最低
  - 低清晰度
- 导出完成判断：
  - download 按钮出现。
  - completed/done/ready 文案。
  - 进度条消失。
  - 生成产物卡片出现。
- 删除产物判断：
  - Delete / Remove / Trash / 清除 / 删除。
  - 删除后产物卡片消失或列表为空。

### 4.4 登录与安全挑战

通用规则：

- OAuth、Sign in、Log in、登录、Google/Microsoft/Apple 登录不应归为 selector miss。
- Cloudflare、人机验证、captcha、verify you are human 不应进入脚本修复。
- 应进入人工等待，并在认证清除后从失败点继续。

### 4.5 视觉规则

视觉 fallback 只替代 click，不替代：

- input
- upload
- drag
- set_range
- set_timecode
- keyboard

视觉回写 selector 必须满足：

- 坐标在截图范围内。
- 可见标签与目标等价。
- DOM 候选唯一或有强证据。
- 点击后状态发生预期变化。
- 没有被 overlay、隐藏元素、邻近控件误导。
