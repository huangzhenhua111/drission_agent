# Web 自动化脚本生成 Agent：当前执行计划书

更新时间：2026-06-23

本文件是后续开发的主计划书。旧文件 `交接完成后的计划书.md` 只作为历史背景和细节归档；接下来推进、验收、复盘都以本文件为准。

---

## 1. 当前目标

把系统从“能在失败后补救、按长计划硬执行”的 Agent，推进成“先列出要完成的事项，再边观察边决定下一步，执行前按风险分流，执行后能验证和沉淀”的脚本生成系统。

最终交付仍按 PDF 要求分成三大模块：

1. `Generation`：从自然语言任务生成可执行脚本。
2. `Debugger`：运行、诊断、修复生成脚本。
3. `Resilience / 容错模块`：为 Generation 和 Debugger 提供可复用的健壮执行、等待、恢复、验证能力。

内部工程实现按三层职责理解：

1. 任务理解与行动规划层：理解用户任务，拆成有顺序的待完成事项 / milestones，维护目标状态；不要一次性假装知道新网站所有具体按钮。
2. 执行与感知层：读取 DOM、截图、候选元素、执行点击/输入/上传/等待，并观察页面状态。
3. 恢复、纠错与沉淀层：失败分类、局部修复、checkpoint、脚本 patch、把成功动作沉淀为稳定脚本。

对外讲三模块；对内按三层职责实现。`app/recovery/` 是三模块共享的内部恢复内核，不是第四个交付模块。

## 3. 当前最核心的问题

现在系统的主要矛盾不是“完全不会点”，而是：

```text
它经常把文本 LLM 预先生成的长计划当成真实页面流程，然后在不确定时先执行，失败后再恢复。
```

这在简单表单页面还能接受，但在视频编辑器里很危险。一次错误点击可能：

- 打开错误弹窗；
- 选错素材；
- 改变时间线焦点；
- 触发错误上传/导出流程；
- 污染后续页面状态；
- 让 replanner 在脏现场上继续做错判断。

所以当前主线改为：

```text
从“长计划硬执行 + 事后恢复”升级为“事项队列 + 滚动观察决策 + 事前避险”。
```

目标流程：

```text
用户任务
  -> 文本 LLM 只拆出有顺序的待完成事项 / milestones
  -> 进入当前事项
  -> 观察当前页面 DOM + screenshot + URL + 已完成事项
  -> 判断当前页面离当前事项还差什么
  -> 生成或筛选下一步候选动作
  -> 风险分流：
       低风险、DOM 高置信：直接 selector 执行
       中高风险、DOM 不确定：问视觉模型选择 DOM candidate_id
       无可靠 DOM 映射：最后才坐标 fallback
  -> 执行动作
  -> typed postcondition 验证当前事项是否完成
  -> 完成则进入下一事项；未完成则继续观察并滚动决策
  -> 失败时先局部修复，必要时 checkpoint replay / replan 剩余事项
  -> 成功动作沉淀为稳定脚本
```

这意味着：文本 LLM 的 plan 不应该是“具体点哪个按钮的长链条”，而应该是“有哪些事项要按什么顺序完成”。具体页面上下一步点什么，必须在运行时根据真实 DOM 和截图决定。

## 4. 设计原则

### 4.1 文本 LLM 只负责事项分解，不负责预言新网站的具体按钮

对新网站，文本 LLM 不可能提前知道真实页面里的每个入口、弹窗、上传流和编辑器控件。因此第一版计划只允许它输出：

```json
{
  "task": "完成用户任务",
  "milestones": [
    "进入目标工具或编辑器",
    "上传指定文件",
    "确认素材进入正确区域",
    "完成编辑操作",
    "导出或保存结果"
  ],
  "success_assertions": []
}
```

它不应该在没看到页面的情况下写死：

```text
点击 Get started -> 点击 Upload -> 点击 Add to timeline -> ...
```

这些具体动作必须由运行时的观察、候选生成、风险判断和视觉确认共同决定。

### 4.2 边点边想下一步，而不是照长计划点到底

每执行一个动作后，都要重新观察页面并更新当前事项状态：

```text
observe -> decide next small action -> risk gate -> execute -> verify -> observe again
```

如果当前事项还没完成，就继续在当前事项内滚动决策；如果已完成，才进入下一个事项。不要因为原始计划里写了后续步骤，就在页面状态不匹配时继续盲点。

### 4.3 滚动规划必须携带 ExecutionMemory，不能每步失忆

滚动规划不是每一步只看当前 DOM/截图重新猜，而是必须维护一个持续更新的上下文账本。每次 text LLM、视觉模型、risk gate、replanner 做决策时，都必须知道：

- 原始任务是什么；
- 事项队列是什么；
- 哪些事项已经完成，完成证据是什么；
- 当前正在推进哪个事项；
- 当前页面是从哪里来的，是哪个动作导致的；
- 当前页面观察结果是什么；
- 下一步大概要服务什么目的；
- 哪些动作不要重复，哪些目标不能弱化。

建议内部结构：

```json
{
  "original_task": "上传 sample.mp4，剪掉前 2 秒，添加标题并导出",
  "milestones": [
    {
      "id": "enter_editor",
      "goal": "进入可编辑工作区",
      "status": "done",
      "evidence": {"action_ids": [0, 1], "postcondition": "editor_ready"}
    },
    {
      "id": "upload_video",
      "goal": "上传 sample.mp4",
      "status": "done",
      "evidence": {"uploaded_file": "sample.mp4", "action_ids": [2]}
    },
    {
      "id": "put_video_on_timeline",
      "goal": "确认视频进入时间线",
      "status": "in_progress",
      "required_postcondition": "timeline_clip_present(video)"
    }
  ],
  "current_milestone": "put_video_on_timeline",
  "navigation_trace": [
    {
      "from_url": "https://example.com/video-editor",
      "action": "clicked visual-confirmed candidate: Get started",
      "to_url": "https://example.com/project/123",
      "transition_goal": "enter_editor"
    }
  ],
  "last_observation": {
    "url": "https://example.com/project/123",
    "title": "Editor",
    "page_kind": "editor_workspace | media_library | upload_dialog | export_dialog | auth | blocked | unknown",
    "page_kind_confidence": 0.72,
    "visible_summary": "...",
    "top_dom_candidates": []
  },
  "next_intent": "place uploaded sample.mp4 onto the timeline",
  "avoid": ["upload sample.mp4 again", "export before timeline_clip_present(video) passes"]
}
```

这个账本的作用：

- 防止已经上传过文件后又重复上传；
- 防止进入项目页后又回头点 landing 页入口；
- 防止导出前还没完成时间线目标；
- 让视觉模型知道它不是自由看图，而是在服务当前事项；
- 让 replan 只围绕当前或剩余事项，不重写已完成 prefix。

### 4.4 低风险直接 DOM，中高风险问视觉

动作执行前必须先做风险分流：

- 低风险：DOM candidate 唯一、selector 稳定、动作可逆或失败代价低，例如普通 input、select、明确唯一按钮。
- 中高风险：候选接近、泛词按钮、上传/插入/导出/项目初始化、modal、timeline、media card、canvas/preview 等区域。

执行策略：

```text
低风险 + DOM 高置信
  -> 直接 selector 执行

中高风险 或 DOM 低置信
  -> 问视觉模型从 DOM candidates 中选择 candidate_id
  -> 用该 candidate 的 selector 执行

没有可靠 DOM candidate
  -> 坐标 fallback
  -> 坐标不能伪装成 selector
```

### 4.5 规则只判断“有没有把握”，不要写死网站语义

代码里的规则不应该承担“这个网站的工作区长什么样”的最终判断。规则只能作为置信度门控：

```text
我有强 DOM 证据 -> 可以直接做
我没有强 DOM 证据 -> 问视觉
```

允许使用通用强证据，例如：

- 真实 file input；
- DOM candidate 唯一且 label/role/action 都匹配；
- typed postcondition 已经通过；
- URL/DOM/state 明确进入项目页。

不允许把某个网站的词表当成普适智能，例如：

- 只因为页面出现 timeline/canvas/my files 就断定一定是可上传工作区；
- 只因为按钮叫 Get started / Start editing 就直接认定该点；
- 为某个网站写死入口按钮链条。

如果规则没有绝对把握，就进入视觉确认。

### 4.6 视觉不是最终执行方式

视觉模型只在 Generation / Debugger 阶段帮助确认动作，不得进入最终脚本。

最终脚本必须是纯 DrissionPage 脚本，不依赖 LLM、视觉模型或 Agent 内部服务。

### 4.7 视觉优先选 DOM candidate，不自由返回坐标

默认流程：

```text
DOM candidates with rect
  -> 截图画编号框
  -> 视觉模型选择 candidate_id
  -> 系统点击该 candidate 的 selector
```

只有完全没有可靠 DOM 映射时，才允许坐标 fallback。坐标 fallback 不能伪装成稳定 selector。

### 4.8 不确定时先避险，不盲点

遇到以下情况，不应直接点击：

- top1/top2 分差很小；
- 多个候选文本相似；
- 目标是 `Start / Create / Continue / Export / Add / Insert` 这类泛词；
- 候选位于 canvas、timeline、preview、media card、overlay、dialog 等高风险区域；
- 点击失败代价高，例如上传、插入时间线、导出、项目初始化。

### 4.9 postcondition 失败优先 local repair

失败后优先修当前 step：

```text
refresh/requery candidates
force vision confirmation
insert missing prerequisite
increase targeted wait
cleanup current dirty state
retry current step
```

只有页面流程已经明显偏离，才 replan remaining goals。

### 4.10 replan 不允许重写已验证 prefix

已经通过 typed postcondition 的动作 prefix 必须进入 GoalLedger。

replan 只能围绕未完成目标生成剩余步骤，不能删除、弱化或重写已经完成的目标。

### 4.11 不能把弱成功当真成功

以下情况不算完整通过：

- capture 成功但 generated script 不能独立 replay；
- CLI 退出码为 0 但关键 postcondition 缺失；
- 素材库出现文件名就当作进入时间线；
- Debugger 通过删除断言、跳过目标、弱化任务来“修复成功”；
- blocked 场景没有明确分类为 auth / quota / security / environment constraint。

## 5. 当前执行路线

### 阶段 1：把长计划改成事项队列 + 滚动决策

这是当前新架构的第一原则。

目标：

- 文本 LLM 先把用户任务拆成有顺序的待完成事项，而不是提前写死完整点击链条。
- 新增 ExecutionMemory / ExecutionContext，持续记录任务目标、事项状态、页面来源、执行历史、当前观察、下一步意图和 avoid 约束。
- Runtime 每次只围绕当前事项决定下一小步。
- 每一步执行后重新观察 DOM + screenshot + URL + 已完成事项，再判断下一步。
- 当前事项完成后才进入下一事项。

计划输出形态应从：

```json
{
  "steps": [
    {"type": "click", "target": "Get started"},
    {"type": "upload", "target": "Upload video"},
    {"type": "click", "target": "Add to timeline"}
  ]
}
```

改成：

```json
{
  "milestones": [
    {
      "id": "enter_editor",
      "goal": "进入可编辑工作区",
      "done_when": ["页面可执行上传或编辑操作"]
    },
    {
      "id": "upload_video",
      "goal": "上传指定视频文件",
      "done_when": ["视频素材已被网站接收"]
    },
    {
      "id": "put_video_on_timeline",
      "goal": "确认视频进入时间线",
      "done_when": ["timeline_clip_present(video)"]
    }
  ]
}
```

运行时循环：

```text
current_milestone
  -> load/update ExecutionMemory
  -> observe page
  -> attach navigation_trace + completed evidence + avoid constraints
  -> infer page state and missing prerequisite
  -> propose next small action candidates
  -> risk gate
       low risk: DOM selector
       medium/high risk: vision choose candidate_id
  -> execute
  -> update ExecutionMemory
  -> verify milestone progress/postcondition
  -> repeat or move to next milestone
```

每次 text LLM / vision / risk gate 的输入都必须包含压缩后的 ExecutionMemory：

```text
original_task
milestones + status
current_milestone
verified completed actions
recent failed/recovered actions
navigation/page transition trace
current observation
next_intent
avoid/repeat constraints
```

建议新增或改造：

```text
app/generation/execution_context.py
```

核心职责：

- 从 planner 输出初始化 milestones；
- 每个动作前生成决策上下文；
- 每个动作后记录 transition 和 evidence；
- 根据 typed postcondition 更新 milestone status；
- 给 planner/replanner/vision/risk gate 输出统一压缩上下文；
- 落盘 `execution_memory.json`，方便复盘和 generated script 沉淀。

验收：

- 对新网站，planner 不需要知道具体按钮名字也能推进；
- artifact 中能看到当前 milestone、页面观察、下一步决策理由、风险分流结果；
- artifact 中能看到 `execution_memory.json`，包括 milestone status、navigation trace、next_intent、avoid constraints；
- text LLM / vision prompt 里都能看到当前事项和已经完成的证据；
- 遇到页面和预期不一致时，不继续执行旧长计划后续步骤；
- 已完成事项不会被重复做，除非 evidence 明确失效；
- replan 只针对当前或剩余事项，不重写已完成事项。

### 阶段 2：修掉真实任务里的弱成功

优先修 Clideo / 123Apps / wide.video 暴露出来的真实问题。

当前重点：

1. image wait 必须生成并验证 `timeline_clip_present(image)`，或合理的 `media_asset_present(image)`。
2. audio wait 必须生成并验证 `timeline_clip_present(audio)`，或合理的 `media_asset_present(audio)`。
3. final confirm 必须验证 video / image / audio 三类媒体都已满足任务目标。
4. 不能只看 `Captured actions` 或进程退出码。

验收：

- 真实网站 capture 成功；
- 关键 wait 都带 typed postcondition；
- generated script 能独立 replay；
- DebugLoop comparison passed；
- 如果 blocked，必须有合理 failure classification。

### 阶段 3：升级 SelectorGrounder 输出协议

让 `SelectorGrounder` 正式输出：

```json
{
  "candidate_id": "e12",
  "selectors": [],
  "candidate": {},
  "selector_metadata": [],
  "score": 320,
  "top_candidates": [],
  "score_margin": 8,
  "confidence": 0.61,
  "reason": "selected candidate e12 with score 320"
}
```

第一版不要求数学完美，只要能稳定区分：

- 明显唯一；
- 候选接近；
- 泛词高风险；
- 无可靠候选。

验收：

- 不改变当前点击行为；
- `generation_trace.json` 记录 top-k、margin、confidence；
- 用真实 artifacts 检查风险信号不过度报警、不明显漏报。

### 阶段 4：实现 PreActionGate + 事前视觉确认入口

新增内部组件：

```text
app/recovery/pre_action_gate.py
```

职责：

- 输入 step、页面 state、DOM candidates、已完成动作；
- 判断当前动作是否有足够证据直接执行；
- 对上传、项目初始化、候选接近、泛词按钮等高风险动作做事前避险；
- 没有强证据时，不盲点，转给视觉模型确认 DOM candidate。

输出示例：

```json
{
  "status": "ready | needs_prerequisite | needs_vision",
  "confidence": 0.84,
  "intent": "enter_upload_capable_editor_workspace",
  "reason": "upload workspace is not proven by DOM; ask vision before uploading"
}
```

当前已完成：

- 新增 `app/recovery/pre_action_gate.py`。
- upload 前只有发现真实 file input / 强工作区证据才直接执行。
- 如果页面像 landing/marketing 或 workspace 证据不足，会先截图问视觉选择 prerequisite DOM candidate。
- prerequisite 执行后不会消费原始 upload step，而是保留原步骤重试。

验收：

- trace/artifact 中能看到 pre-action gate 判断；
- marketing 页不能被误判成 upload-capable editor workspace；
- 真正有 file input / 强工作区证据时不能无意义调用视觉；
- prerequisite 点击后原始 upload 仍会重试；
- 后续继续把 click/export/add/timeline/media card 等动作纳入同一套 gate。

### 阶段 5：中高风险 click 启用视觉 candidate_id 确认

新增或完善：

```text
app/generation/visual_fallback.py::VisualCandidateConfirmer
```

职责：

- 把 top-k DOM candidates 与截图一起发给视觉模型；
- 调用视觉模型；
- 只允许返回 candidate_id；
- 验证 candidate_id 在候选集合内；
- 用对应 selector 执行点击。

坐标 fallback 单独走 schema，不能和 candidate 确认混在一起。

当前已完成：

- 新增 `VisualCandidateConfirmer` 和 `VisualCandidateChoice`。
- 视觉确认只接受候选集合内的 `candidate_id`，低置信度或未知 id 会被拒绝。
- `CaptureRunner` 在 `AmbiguousTargetError` 时优先问视觉选择 DOM candidate，而不是立刻 replan。
- 视觉选中的 candidate 会用 selector 执行，并记录到 `visual_candidate_confirmations/step_XX.json`。

验收：

- Open Sans / Text sidebar 等 DOM 不充分控件，视觉能确认正确 candidate；
- Text sidebar 不能误回写成附近的 Text to speech；
- canvas 上无 DOM 对象时允许保留纯坐标；
- 点击后页面无变化时不能宣称回写成功；
- 多层嵌套 span/div 应回写真正可点击父节点。

下一步扩展：

- SelectorGrounder 输出统一的 top-k / margin / confidence，方便更多不确定场景直接进入视觉确认。
- 对 Export、Add/Insert、timeline/media card、modal confirm 等高风险 click 接入同一套视觉 candidate 确认。
- 需要时给截图叠 candidate 编号框，降低视觉模型在复杂页面上的候选对应成本。

### 阶段 6：postcondition 失败后 local repair 优先

失败时先按当前 step 局部修复，而不是直接 full replan。

新增或细化 RecoveryDecision：

```text
VISION_CONFIRM_CANDIDATE
LOCAL_REPAIR_CURRENT_STEP
RESTORE_CHECKPOINT
REPLAY_VERIFIED_PREFIX
INSERT_PREREQUISITE_THEN_RETRY
REPLAN_REMAINING_GOALS
```

决策顺序：

```text
if current step not vision-confirmed:
    cleanup/restore
    force vision_confirm_candidate
    retry current step
elif selector stale:
    refresh_requery current step
elif prerequisite missing:
    insert prerequisite current step
elif postcondition too early:
    targeted wait
elif page flow changed:
    replan remaining goals
else:
    stop or escalate
```

验收：

- postcondition_failed 不再默认直接 replan；
- local repair 的过程有 artifact；
- 已完成目标不会被后续修复删除或弱化。

### 阶段 7：实现 replayable checkpoint

新增：

```text
app/recovery/checkpoint.py
```

checkpoint 不做虚假的浏览器内存快照，只保存可重放 verified prefix。

artifact：

```text
checkpoints/checkpoint_XX.json
checkpoints/checkpoint_XX.png
checkpoints/replay_prefix_XX.json
```

失败后三档恢复：

```text
轻量恢复：Esc / 关闭弹窗 / page.back()
中度恢复：reload 当前 URL，再验证已完成目标
重度恢复：新页面或新浏览器上下文，从 verified actions replay 到上一步
```

验收：

- 每个 required postcondition 通过后能保存 verified prefix；
- 脏状态后能 replay 到最近有效 checkpoint；
- replay 后必须重新验证已完成目标。

### 阶段 8：完成 DebugLoop 闭环

闭环：

```text
运行 generated script
  -> 收集 failure context
  -> FailureClassifier
  -> ScriptFixer 局部 patch
  -> 保存 patched_script_01.py
  -> replay + postcondition 验证
  -> 成功 / 下一轮 / 升级给 LLM / 判定不可满足
```

每轮必须留下：

```text
debug_attempt_01.json
failure_context_01.json
patch_01.diff
patched_script_01.py
script_run_01.json
```

验收：

- generated script 不是只靠 capture 成功；
- patch 不能删除或弱化任务目标；
- 最终输出 `script_final.py`；
- performance comparison passed。
